# Project: instantly-service

Cold email outreach service via Instantly.ai API V2. Handles campaign management, lead upload, email accounts, warmup, analytics, and webhook processing.

## Commands

- `npm test` — run all tests (Vitest)
- `npm run test:unit` — unit tests only
- `npm run test:integration` — integration tests only
- `npm run build` — compile TypeScript + generate OpenAPI spec
- `npm run dev` — local dev server (nodemon)
- `npm run generate:openapi` — regenerate openapi.json
- `npm run db:generate` — generate Drizzle migrations
- `npm run db:migrate` — run migrations
- `npm run db:push` — push schema to database
- `npm run backfill:inferences` — one-shot CLI: project synthetic predecessor events onto existing silver rows. Manual only — NEVER wired into boot (port-bind hazard).
- `npm run cleanup:stacked-sigs` — one-shot CLI: clean stacked signatures from Instantly campaigns whose lead has been pushed but not yet received any email. Dry-run by default; pass `-- --commit` to actually PATCH Instantly. See [Signature handling](#signature-handling--idempotent-strip-then-append). Manual only.
- `npm run audit:dupes -- <orgId>` — read-only CLI: audit cross-campaign duplicate contacts (same email sitting in ≥2 **active** Instantly campaigns). **Source of truth = Instantly API**, never the local DB (the DB can be stale — the duplicate FACT is derived from Instantly). The local DB is read only to label each collision `same-brand` / `cross-brand` / `unknown-brand` (Instantly has no brand field). Read-only — audits, never heals. Flags: `--json`, `--limit N`, `--severe-only`. Manual only. See [Cross-campaign duplicate audit](#cross-campaign-duplicate-audit--read-only).
- `npm run heal:dupes -- --commit` — MUTATING CLI: pause redundant active Instantly campaigns. For each email in ≥2 active campaigns, **keep the oldest** (`timestamp_created`) and **pause** the rest. Dry-run by default; `--commit` to act; `--limit N` to batch. Pause is reversible (no delete). Instantly-only (no cost refund / DB write — runs-service is internal-DNS-only from a local shell). Run AFTER the retry-stuck fix deploys (else the worker re-spawns dupes). Idempotent/resumable. Manual only. See [Cross-campaign duplicate audit](#cross-campaign-duplicate-audit--read-only).
- `npm run cleanup:finished-contacts -- --commit` — MUTATING CLI: delete contacts of **finished** Instantly campaigns to reclaim the plan's uploaded-contacts quota. Finished = Instantly status **paused (2)** or **completed (3)**; active (1) is never touched. **Source of truth = Instantly API** (status + membership), never the local DB. `DELETE /leads` is campaign-level (the only delete that frees quota — deleting from "Lists" does not). Dry-run by default; `--commit` to act; `--limit N` to batch; `--json`. Idempotent/resumable (already-deleted leads drop out of the next sweep; a 404 delete is counted, not fatal). Instantly-only (no local DB write — the recurring reconcile path marks rows terminal). Selection logic is pure + unit-tested in `src/lib/cleanup-finished.ts` (`selectContactsToDelete`). Manual only. See [Finished-contact cleanup](#finished-contact-cleanup--reclaim-quota).

## Architecture

- `src/index.ts` — Express app entry point
- `src/schemas.ts` — Zod schemas (source of truth for validation + OpenAPI)
- `src/zod-setup.ts` — Side-effect module that extends Zod with `.openapi()`. Import it BEFORE any module that creates `z.object(...).openapi("Name")` schemas.
- `src/routes/` — Route handlers (campaigns, leads, accounts, analytics, send, webhooks, health)
- `src/middleware/serviceAuth.ts` — API key auth middleware
- `src/lib/instantly-client.ts` — Instantly API V2 HTTP client
- `src/lib/runs-client.ts` — Runs service HTTP client for cost tracking
- `src/db/schema.ts` — Drizzle ORM table definitions (campaigns, leads, accounts, events). Gold stats are derived-on-read from silver in `analytics.ts`/`status.ts` — there is NO `analytics_snapshots` materialized table (a dropped legacy table; do not reintroduce a snapshot **table** cache). Live aggregation was proven too slow under load (2026-06-13: `/orgs/stats` + `/public/stats` flooded email-gateway with ~10s AbortSignal timeouts — bursts of identical re-aggregations saturating the 0.25-1 CU Neon compute). The mitigation is the LIGHTEST possible, NOT a snapshot table: (1) a short-TTL in-memory cache (`src/lib/stats-cache.ts`, 60s default, per-replica) on GET `/stats`, GET `/public/stats`, **POST `/orgs/status`** (v0.57.2 — keyed on orgId+brandId+campaignId+sorted emails), AND **GET `/internal/audit/sending-forecast`** (v0.60.2 — platform-scoped, fixed key `sending-forecast`) — keyed on params (+ orgId for the authed routes). **RULE: any route that live-aggregates over the silver event log AND is called by email-gateway on the send/retry path MUST wrap its DB work in `getOrSetCachedStats`.** email-gateway retries the SAME request on its ~10s AbortSignal timeout and the abort does NOT cancel the server-side query, so each retry piles another identical heavy aggregation onto the pool (max 20) → the timeout flood. The in-flight dedup collapses the retry storm to one query. TWO routes have been the gap that bit — audit EVERY email-gateway-hit aggregation route, not just `/stats`: (a) `/orgs/status` was uncached, then #426 (contract v1.2.0) added a per-scope `COUNT(DISTINCT step)` sentCount subquery to it → the flood returned on `POST /orgs/status` 11h after that merge (fixed v0.57.2); (b) `/internal/audit/sending-forecast` was uncached (its `loadPendingLeads` fleet scan + a per-lead config LATERAL, plus a live paginated `listAccounts`) → email-gateway (`/public/stats/sending-forecast`) → features-service → api-service all 500'd (fixed v0.60.2, #464 — cache wrap + a composite index, see the [Sending forecast](#sending-forecast--capacity-vs-scheduled-volume) section). Before adding ANY new aggregation to a status/stats/forecast route, confirm the route is cached; (2) `instantly_events_stats_covering_idx` (campaign_id, event_type, lead_email, step) → index-only scan on the filtered `/orgs/stats` path (does NOT help the no-filter `/public/stats`, which the planner seq-scans regardless — that path leans on the cache); (3) `queryStats` / `queryGroupedStats` / the per-step block (`computeStepStats`/`computeStatsPayload`) parallelize their independent reads via `Promise.all`; (4) `work_mem = '32MB'` set per pool connection (`src/db/index.ts`, `pool.on("connect")`). Each `COUNT(DISTINCT lead_email)` sorts ~88k events; at the pg default (4MB) that sort spills to temp disk (`EXPLAIN`: `external merge Disk: 7616kB`) → I/O contention + the pool connection held longer, which is what starves the pool (max 20) under burst and produces the `timeout exceeded when trying to connect` flood. At 32MB the sort stays in memory (`quicksort Memory: 11312kB`, zero temp). RAM-safe: the compute autoscales to 1 CU/4GB under exactly this load; the 0.25 CU idle floor runs no such sorts. Do NOT lower `work_mem` back to the default to "save RAM" — it reintroduces the spill. The CU floor was deliberately NOT bumped (cost); `work_mem` buys headroom, not infinite scale — if the flood returns under higher traffic, the 1 CU ceiling is the next wall. If you add a new cached stats route, `clearStatsCache()` MUST be called in its unit-test `beforeEach` or cross-test cache hits skip the DB and break assertions.
- `src/db/index.ts` — Database connection. Pool tuned for the Singapore Neon compute under stats burst: `max:20`, `idleTimeoutMillis:60_000`, `keepAlive:true`, `connectionTimeoutMillis:5_000` (was 15_000 — a starved pool-acquire blocked the next request the full 15s before `withConnectRetry` retried; 5s fails fast → retry, still covers a staging/dev cold Neon resume), and `work_mem='32MB'` per connection (see the stats note above). Do NOT raise `connectionTimeoutMillis` back to 15s.
- `src/routes/audit.ts` — Staff ops reads/writes (platform-scoped, no org). `GET /internal/audit/sending-forecast` powers admin.distribute.you's "Audit → Instantly" page: fleet **capacity** vs upcoming scheduled **volume** in one response (see [Sending forecast](#sending-forecast--capacity-vs-scheduled-volume)). `POST /internal/audit/accounts-sync` snapshots `GET /accounts` → bronze/silver + runs `reconcileLifecycle` (202 + background, `accounts-sync: done`; wired as the placement cron's first step). `GET /internal/audit/capacity-history?days=N` (clamped 1–365) → `{ series: [{ date, inProductionCount, dailyCapacity }] }` reconstructed from `instantly_accounts_raw` (daily_limit over time) + lifecycle events (status over time). See [Account lifecycle](#account-lifecycle--auto-driven-per-account-state-bsg).
- `src/lib/account-lifecycle.ts` — pure `deriveLifecycle` (4-state, first-match-wins). `src/lib/account-lifecycle-sync.ts` — `reconcileLifecycle` IO glue (PATCH warmup on flip → insert lifecycle event → update silver).
- `tests/unit/` — Unit tests
- `tests/integration/` — Integration tests
- `tests/helpers/` — Test utilities (test app, test DB)

## Sending forecast — capacity vs scheduled volume

`GET /internal/audit/sending-forecast` (platform-scoped, no org; same `serviceAuth` tier as `/internal/accounts`). Pure logic in `src/lib/sending-forecast.ts` (unit-tested `tests/unit/sending-forecast.test.ts`); route `src/routes/audit.ts` does IO only. Two independent halves in one response:

**Performance (v0.60.2, #464) — the handler is CACHED and the config LATERAL is INDEXED. Do NOT unwrap either.** (1) The whole computation is wrapped in `getOrSetCachedStats` (60s TTL, fixed key `sending-forecast`) — this route lives-aggregates AND makes a live paginated `listAccounts` call, and email-gateway (`/public/stats/sending-forecast`) retries on its ~10s AbortSignal timeout without cancelling the server query, so the uncached version flooded the pool → features/api 500s (same class as `/orgs/status`; see the `src/db/schema.ts` cache RULE). The in-flight dedup collapses the retry storm to one loader run. Fail-loud preserved (a loader throw caches nothing, still 500s). `clearStatsCache()` is in `audit-route.test.ts` `beforeEach`. (2) `loadPendingLeads`'s per-lead latest-config LATERAL (`WHERE instantly_campaign_id=$1 ORDER BY fetched_at DESC LIMIT 1`) is served by the composite index `instantly_campaigns_config_raw(instantly_campaign_id, fetched_at DESC)` (migration `0026`) — ordered index scan + LIMIT 1 reading only the winning row, vs the old bitmap-heap-scan-all-~16-configs + top-N-sort of their large jsonb payloads (~50k block reads/call × 6,400 loops; prod EXPLAIN 678ms→349ms warm, far larger cold). **The table is ~550k rows / ~1.38GB, so migration `0026` is `CREATE INDEX IF NOT EXISTS` and the index was pre-built CONCURRENTLY out-of-band** (a non-concurrent boot build scans the whole heap → port-bind outage; boot-window rule). Any new prod-scale env MUST pre-build it concurrently before deploy.

- **Capacity** (`dailyCapacity`, `healthyAccountCount`, `totalAccountCount`, `blockedDomainCount`): `computeCapacitySummary(accounts, lifecycleByEmail)`. `dailyCapacity = Σ daily_limit` over the **`in_production` accounts** (the exact live-send pool — `lifecycle_status == 'in_production'`, read from silver; see [Account lifecycle](#account-lifecycle--auto-driven-per-account-state-bsg)). `blockedDomainCount` = the count of `deactivated_by_user` accounts (the `instantly_domain_policy` brand/product domains). Under the 2026-06-29 Gmail-spam halt nearly the whole fleet is `in_recovery`, so `dailyCapacity` is legitimately near-zero — render it, never fabricate a non-zero.
- **Volume** (`days[]`): `projectDailySchedule(pendingLeads, asOf)`. A TRUE per-day projection of active campaigns' remaining **un-sent** steps onto future calendar days, **capacity-INDEPENDENT** and bounded by sequence structure. It is NOT `backlog ÷ capacity` — with capacity ~0 a capacity-paced drain would never terminate. Source of "un-sent" = `sequence_costs` rows with `status='provisioned'` (a hold stays provisioned until its `email_sent` webhook actualizes it), gated to live campaigns (`instantly_campaigns.status='active'` AND `delivery_status IN ('contacted','sent')`).
- **Cadence = the REAL per-step delays from bronze config (shared with the queue breakdown, v0.57.6+).** `loadPendingLeads` resolves each lead's latest bronze sequence-config step delays (`instantly_campaigns_config_raw.payload->'sequences'->0->'steps'->i->>'delay'`) and passes them as `PendingLead.stepDelays` (config-ordered 0-based; gap cost-step `k`→`k+1` = `steps[k-1].delay`, identical indexing to `queue-breakdown.ts`). `scheduleLead` accumulates the real per-gap delays; `STEP_GAP_CALENDAR_DAYS = 3` is now the per-gap FALLBACK for a missing config delay (also the queue-breakdown fallback → one shared fallback). This makes the forecast's inter-step cadence identical to the per-account queue-bucket projection — the two ops views share ONE cadence source of truth. Guard: the `delayForGap` / real-delay `scheduleLead` / `classifyQueuedSequence`-coherence tests in `tests/unit/sending-forecast.test.ts`. Do NOT revert to a hardcoded canonical gap. **Day bucketing = the raw nominal UTC send day, with NO weekend snap — IDENTICAL to the queue breakdown, and load-bearing for coherence.** The former weekend forward-snap (business-hours-weekday model) pushed a weekend-due step to the next Monday, so on a weekend today/tomorrow read `scheduledCount:0` while the per-account "Sending accounts" table (which never snaps) still counted the step as `queuedNextToday`/`queuedNextTomorrow` — the two staff surfaces disagreed for the same pending steps. Dropping the snap makes them agree by construction: a weekend day carries its REAL due-step count, and the forecast's per-day count reconciles with the fleet-aggregated queue-breakdown buckets. Do NOT reintroduce a weekend snap. Guard: the "raw nominal day (NO snap)" + weekend-asOf-AC + "weekend day carries its REAL due-step count" tests in `tests/unit/sending-forecast.test.ts`. Same honesty caveat as the breakdown: the projected date is a nominal-cadence LOWER BOUND (real dispatch slips later under daily-limit/throttling), not Instantly's exact dispatch date.
- **Key resolution:** the shared cold-email workspace key comes from **key-service's platform decrypt endpoint** `GET /keys/platform/instantly/decrypt` (global platform key, no org/user — `resolvePlatformInstantlyApiKey` in `key-client.ts`), NOT a local env var. Do NOT reintroduce a `process.env.INSTANTLY_API_KEY` read on the request path (that's a CLI-only override). 404 (platform key not configured) → fail loud 500, no impact on other routes. Fail-loud throughout — no silent zero fallbacks.

## Reconciliation audit — local truth vs Instantly truth, side by side

`GET /internal/audit/reconcile` (platform-scoped, no org; same `serviceAuth` tier as `/internal/audit/sending-forecast`). Pure logic in `src/lib/reconcile-audit.ts` (unit-tested `tests/unit/reconcile-audit.test.ts`); route `src/routes/audit.ts` does IO only. For each **countable fact** returns `{key, label, local, instantly, delta, sourceOfTruth}` where `delta = local − instantly` — so an operator seeing a non-zero delta on admin.distribute.you can investigate the underlying bug (lost webhook, lagging reconcile, a pause/throttle we never observed).

- **Reconciles COUNTS, never the forecast's future DATES.** The sending-forecast VOLUME is an inherent projection — the exact future send DATE of a step exists nowhere (Instantly schedules dynamically off daily limits / business hours / throttling / pauses), so timing cannot be reconciled. Do NOT add a "date reconciliation" — it would compare against a number that doesn't exist. (The forecast's volume MAGNITUDE, however, IS reconcilable — see `pendingSends` below.)
- **Output = THREE comparable metrics only (`activeCampaigns`, `emailsSent`, `pendingSends`); Instantly is SoT for all three.** `buildReconciliation` emits exactly these. TWO former rows were REMOVED (v0.50.7) because their two sides measure DIFFERENT quantities, so their delta is a structural artifact, not drift an operator can act on — do NOT re-add them to the output:
  - `contactedDispatched` ("Leads dispatched"): Instantly's fleet-aggregate `contacted_count` tracks per-step DISPATCHES (empirically ≈ `emails_sent_count`), NOT distinct leads, while the local side counts distinct `(campaign, lead)`. Proven airtight against prod (68,570 contacted ≈ 69,295 emails — impossible under a distinct-leads reading, since a contacted lead has ≥1 email). Distinct-leads vs dispatch-count always shows a huge fake gap.
  - `contactsStored`: local counts ALL `instantly_campaigns` rows ever created (never decrements), while Instantly's `leads_count` is CURRENTLY-stored and drops as the armed finished-contact cleanup (`DELETE_FINISHED_CONTACTS_ENABLED=true`) deletes contacts to reclaim quota. Guaranteed-positive-by-design, not drift.
  - Both counts are STILL computed (`summarizeInstantlyCounts` + the local SQL + the snapshot columns) — only the output pairing was dropped. The snapshot table keeps its 5 columns (no migration).
- **`pendingSends` counts STEP-SENDS on both sides, and is DERIVED on Instantly.** Instantly has no ready-made remaining-step field, but each campaign's `sequences` carry the loaded followup steps and analytics carry `emails_sent_count`, so (one campaign = one lead) Instantly's remaining = `Σ over ACTIVE campaigns of max(0, stepCount − sent)` (`computeInstantlyPendingSends`). Do NOT re-mark `pendingSends` local-only — the data is derivable, not missing (regression guard: it was shipped local-only in v0.50.4, corrected in the next hotfix). **UNITS (v0.50.7 fix): the local side counts `COUNT(DISTINCT (campaign_id, lead_email, step))` provisioned holds, NOT `COUNT(*)`.** Each step provisions TWO `sequence_costs` rows (account + domain), so `COUNT(*)` DOUBLES the local number vs Instantly's one-per-step remaining. Do NOT revert to `COUNT(*)`.
- **Instantly side = two bulk sweeps.** (1) `listAllCampaignAnalytics` → `GET /campaigns/analytics` with `id` omitted returns ALL campaigns' analytics as a single **non-paginated** array (confirmed via developer.instantly.ai — no cursor/limit on this endpoint), the ONE client helper that deliberately does NOT paginate. (2) `listAllCampaignSequenceLengths` → paginated `GET /campaigns` (each campaign object carries its `sequences`) for the per-campaign `stepCount`. Per-campaign iteration is infeasible at fleet scale (thousands of campaigns × rate limit); both sweeps are bulk/paginated, not N calls. Instantly ACTIVE lifecycle code = `1`.
- **Local side = one round-trip.** Five scalar subqueries in a single `db.execute`. `pendingSends` reuses the EXACT `loadPendingLeads` gate so its number equals the total steps the forecast projects. Read live on the request path (fast).
- **Instantly side = pre-aggregated snapshot, NEVER swept live on the request path.** The two Instantly sweeps (`listAllCampaignAnalytics` + `listAllCampaignSequenceLengths`) are throttled fleet-wide pagination — `listAllCampaignSequenceLengths` paginates `/campaigns` at ~110ms/page across thousands of campaigns → **minutes**, which blew past the gateway/browser timeout and left admin.distribute.you's Audit→Instantly reconcile table on an **infinite loading skeleton** (v0.50.x). The fix pre-computes the five Instantly counts into a single-row snapshot table `instantly_reconcile_snapshot` (migration `0022`, PK = fixed `'singleton'`) via a **background refresh**; the GET reads that row in one fast query (local SQL + snapshot read in `Promise.all`), returning in a few seconds cold. `src/lib/reconcile-snapshot.ts` = IO glue (`refreshInstantlySnapshot` runs the slow sweep + upserts; `readInstantlySnapshot` = fast read; `maybeTriggerRefresh` = guarded fire-and-forget). Do NOT reintroduce the live sweep in the GET handler.
  - **Refresh paths:** (1) `POST /internal/audit/reconcile/refresh` — 202 + background, runs the sweep + upserts. SEED with it right after deploy (so the first dashboard GET is 200) + wire as a cron freshness trigger. (2) On-read **stale-while-revalidate** — the GET serves the snapshot immediately and kicks a background refresh when it is older than `RECONCILE_SNAPSHOT_TTL_MS` (30 min, pure `isSnapshotStale`), guarded by an in-flight flag so concurrent stale reads don't stack sweeps.
  - **Fail loud, never fabricate:** a MISSING snapshot (never computed) → GET **503** + kicks the seeding refresh (no fabricated Instantly zero); a genuine handler error still → **500**. The snapshot values ARE Instantly's numbers (fetched slightly earlier); the local side is live, so `delta` is live-local − recent-Instantly — correct for a drift audit. `asOf` stays request-time.
- **Orphaned-hold repair — `POST /internal/campaigns/actualize-orphaned-sends`** (`src/lib/actualize-orphaned-sends.ts`, mirror of `refund-stranded-holds`). One-time, idempotent, resumable, **in-cluster** sweep. Actualizes `provisioned` `sequence_costs` whose step ALREADY has a real (`inferred=false`) `email_sent` silver event — the send fired but a transient runs-service error left `handleEmailSent`'s actualize swallowed, and reconcile's event-count drift gate never retries it (the event IS present → `remote == local` → no drift). Reuses the exported `handleEmailSent`. **A hold whose run is GONE (runs-service 404 — retention purged the run) can NEVER actualize; `handleEmailSent` now detects that terminal 404 (`isRunGoneError` in `runs-client.ts`, 404-only — NOT 403/5xx) and cancels the hold LOCALLY (`status='cancelled'`, the send fired but is unbillable), so the sweep auto-cleans it under `costsCancelled` instead of re-stranding it (issue #416, was `failed` pre-fix). A TRANSIENT error (5xx/timeout/403) still leaves the hold `provisioned` under `costsTransient` for the next run. Never fabricates.** `handleEmailSent` returns an `EmailSentOutcome` (`{actualized,cancelled,transient}` cost-row counts); the sweep summary is `{stepsProcessed, costsActualized, costsCancelled, costsTransient, stepsFailed}`. `{limit}` bounds the batch. Watch logs for `actualize-orphaned-sends: done`. (~692 rows in prod at v0.50.7; the ~1,752 gone-run backlog was manually DB-cancelled 2026-07-01 — the fix prevents recurrence on the live `handleEmailSent` path.) **NOW WIRED INTO THE DAILY RECONCILE CRON (`reconcile-cron.yml` step 2, 2026-07-11): the sweep is no longer manual-only, so sent-but-unbilled holds self-heal every 03:00 UTC instead of accumulating until someone remembers to trigger it. Idempotent — a run with nothing orphaned no-ops.**
- **A provisioned email-send hold on an Instantly-ACTIVE campaign is CORRECTLY IN-FLIGHT — do NOT read the provisioned-hold backlog as a reconcile bug, and NEVER blanket-cancel or blanket-actualize it (2026-07-11 audit).** `POST /send` provisions ALL sequence steps' holds upfront (`send.ts` loop), and each step-run is marked `completed` at /send time — so **"run completed" ≠ "email sent"** (a step-3 run is `completed` the instant the lead is pushed, long before step 3 dispatches). Real send evidence is a real (`inferred=false`) `email_sent` silver event, which THIS service owns. The upfront provisioning is load-bearing: the ops forecast (`sending-forecast`), queue breakdown, capacity-aware account selection, and account-health `queueSize` ALL read `sequence_costs status='provisioned'` as the set of scheduled-but-un-sent future steps. A hold resolves ONLY when its step's send resolves: `email_sent` → actualize (webhook + this sweep); sequence terminal (reply/bounce/unsub, or Instantly campaign paused/completed) → cancel (`cancelRemainingProvisions` / reconcile finish-closure). Under the 2026-06-29 Gmail-spam halt, Instantly keeps stalled campaigns at status=**active(1)** indefinitely while deferring the later/first sends (accounts `in_recovery` / daily-limit throttled), so their holds have no resolution signal yet — they sit `provisioned` legitimately (prod 2026-07-11: ~22.3k of ~23.4k provisioned holds were on Instantly-active campaigns with config pulled that same day; only 574 were real-sent-but-unbilled orphans, healed by this sweep). Cancelling the active-campaign holds would contradict Instantly (source of truth) and under-count sends that may still fire; actualizing them fabricates a send. Leave them. The backlog trends toward the genuinely-resolvable subset (real-sent orphans + Instantly-terminal cancels), NOT toward 0, as long as the halt keeps sends deferred.
- **Known reconcile-coverage gap (NOT fixed by these deltas — separate follow-up):** the 3 remaining deltas (`activeCampaigns`/`emailsSent`/`pendingSends`) heal via the nightly `reconcileAll`, but it is a fire-and-forget background job with NO time cap, and per-run it covers only ~10k of ~11.3k active campaigns (dies partway on the 0.25 CU Neon compute + `getCampaignAnalytics`-null early-returns write no bronze), leaving a ~2k-campaign tail unhealed for 24-48h. "Extend the run to 2h" is a non-lever (no time budget exists). The real levers are: run the reconcile cron 2-3×/day (union covers the tail), handle the null-analytics early-return, and make the sweep resumable. Tracked separately — do NOT expect these deltas to zero every night.

## Shared contract

Cross-provider canonical shapes (`StatusScope`, `RecipientStats`, `EmailStats`, `StepStats`, `RepliesDetail`, `ChannelStats`, `ProviderStatus`, `GlobalStatus`, `ReplyClassification`) live in [`@shamanic-technologies/email-domain-contract`](https://github.com/shamanic-technologies/email-domain-contract). Do NOT redeclare these schemas locally — re-export from the package via `src/schemas.ts`. Instantly-specific schemas that wrap or extend these shapes keep their local declaration.

Two provider-specific fields are **optional in v1** of the contract: `cancelled` and `notSending`. Instantly-service is the source of truth for both — handlers always emit them as populated values. Contract v2 will tighten them to required once postmark-service ships its padding follow-up.

**Contract pinned at `^1.1.0`** (DIS-229). v1.1.0 widened `StatusScope` with 8 optional+nullable first-occurrence (MIN) timestamps — the mirror of `lastDeliveredAt` (MAX): `firstContactedAt, firstSentAt, firstDeliveredAt, firstOpenedAt, firstClickedAt, firstRepliedAt, firstBouncedAt, firstUnsubscribedAt`. `POST /orgs/status` populates all 8 on every StatusScope (`byCampaign[*]`, `campaign`, `brand`) in `src/routes/status.ts`. Rules: each `first*At = MIN(event timestamp in scope)`, null if it never occurred; brand = MIN across the brand's campaigns (mirrors `BOOL_OR` booleans). Each `first*At` agrees with its boolean (non-null iff boolean true). **Two have no engagement event and are derived:** `firstContactedAt = MIN(c.created_at)` (`contacted` = campaign row exists / lead pushed), and `firstDeliveredAt = MIN(sent-timestamp) when (sent AND NOT bounced) else NULL` (consistent with `delivered`). Don't rename the 8, don't add a 9th — the set is locked to the published contract. Guard: the "First-occurrence (MIN) timestamps — DIS-229" tests in `tests/unit/status.test.ts`.

## Zod 4 caveat — contract schemas + `.openapi()`

`@asteasolutions/zod-to-openapi` attaches `.openapi()` to Zod schema instances at the time `extendZodWithOpenApi(z)` runs in the consumer. The contract package's schemas were instantiated before that point in the consumer's module graph, so they do NOT gain `.openapi()` retroactively. Re-export them without `.openapi(name)` and let the generator inline them (no `$ref` name). Local schemas defined in `src/schemas.ts` (after `import "./zod-setup"`) keep their `.openapi(name)` tagging.

## node-postgres array parsing — return `jsonb_agg`, NEVER a raw `numeric[]`

Any SQL (`db.execute(sql\`…\`)`) that returns a Postgres ARRAY for JS consumption MUST build it as a **JSONB array (`jsonb_agg(...)`)**, NOT a native array type via `ARRAY(SELECT …)` / `ARRAY_AGG(<non-int>)`. **node-postgres parses `jsonb` into a JS value reliably, but returns a `numeric[]` (and other non-`_int4` array OIDs) as its RAW TEXT form — a string like `"{3,7,0}"`** — so `?? []` does NOT catch it (a string is truthy) and a downstream `.map(...)` throws `X.map is not a function`. `int[]` from `ARRAY_AGG(<int column>)` happens to parse fine (pg-types registers `_int4`), which is why the trap hides — a query that works with an int array breaks the moment you switch the element to `numeric`/`text`/computed. Prefer `COALESCE(jsonb_agg(expr ORDER BY …), '[]'::jsonb)`; guard a null/non-array source (`jsonb_array_elements` throws on a non-array) and, in the mapper, `Array.isArray(x) ? x : []` so a shape surprise degrades one row, never 500s the endpoint. The per-account queue breakdown (`account-sending-stats.ts`) selects raw jsonb (`payload->…->'steps'`) and was safe; the sending-forecast loader (`audit.ts` `loadPendingLeads`) shipped a `numeric[]` and 500'd the whole fleet forecast — `rawDelays.map is not a function` (fixed v0.59.1, #454). Pure-function unit tests CANNOT catch this — the mis-parse only happens against real node-postgres, not the mocked `db.execute`; validate array-returning SQL against prod (Neon MCP) before shipping.

## Instantly client — pagination convention

Any helper in `src/lib/instantly-client.ts` that returns "all of X" from Instantly (no caller-driven `skip` / `limit` parameters) MUST paginate via the `next_starting_after` cursor pattern used by `listAccounts`, `listLeadsFull`, and `listEmails`. NEVER call `/path` once and return `response.items` — Instantly's default page size is **10**. Every such helper requires a unit test asserting:

```
{account.first_name} {account.last_name}     ← derived from the sending account
Distribute.you | Marketing Agency            ← fixed SIGNATURE_BRAND_LINE
```

`buildDefaultSignature(account)` joins `account.first_name` + `last_name` (falls back to `DEFAULT_SENDER_NAME = "Kevin Lourd"` when both are absent). **NO title** — a fixed "Founder" cannot apply across many distinct sender personas (DFY pre-warmed accounts ship with names like Amy Moore / Louis Williams), so the title line was dropped 2026-06-30. The brand line renders as **plain text — NOT a clickable `<a>` link**: `buildEmailBodyWithSignature` autolinkifies the prospect body ONLY, appending the signature block verbatim. Do NOT reintroduce sig-wide autolinkify.

The signature is intentionally code-derived (not Instantly-UI driven). When the brand line / format changes, edit `SIGNATURE_BRAND_LINE` / `buildDefaultSignature` and ship a hotfix. Do NOT re-add per-account UI signatures without updating this rule — code prefers the UI value over the derived default. Guard: the "signs with the account's OWN name" + canonical-fallback tests in `tests/unit/send.test.ts`.

**Visible unsubscribe footer (`UNSUBSCRIBE_FOOTER_HTML`, appended below EVERY signature).** `buildEmailBodyWithSignature` appends a small / grey / italic opt-out line — a `<p>&nbsp;</p>` spacer paragraph (the blank-line gap) then `Don't want to hear from me again? <a href="{unsubscribe_link}">unsubscribe</a>` — as part of the `sigBlock` (INSIDE the strip-and-reappend region, so idempotency holds — a re-sent body never stacks the footer). **The link MUST use Instantly's SINGLE-brace `{unsubscribe_link}` server-side merge variable — the `{{...}}` mustache form does NOT resolve** ([help.instantly.ai/articles/6191077](https://help.instantly.ai/en/articles/6191077-how-to-add-unsubscribe-link)); it renders per-lead at send into a functional opt-out URL. This is the VISIBLE opt-out path, distinct from and complementary to the RFC-8058 `List-Unsubscribe` header set on every campaign (`insert_unsubscribe_header: true` in `createAndActivateCampaign`), which powers the mailbox-native (Gmail/Apple/Outlook) one-click unsubscribe button. **Instantly's HTML sanitizer PRESERVES the inline `style` attribute on PATCH/create round-trip — verified 2026-07-19 via a live create→GET→delete probe: `style="font-size:12px;color:#999999;font-style:italic"` and the anchor + `{unsubscribe_link}` survived intact (only `&nbsp;`→space and `<br>`→`<br />` normalization).** So the grey/small/italic rendering is real, not stripped. Because the footer adds one intentional `<a>` anchor, the send tests assert the ONLY anchor is the unsubscribe link (`(result.match(/<a /g)).length === 1` + `href="{unsubscribe_link}"`), NOT `not.toContain("<a ")`. Guard: the "visible unsubscribe footer" + spacer + idempotent-with-footer tests in `tests/unit/send.test.ts`. Do NOT switch `{unsubscribe_link}` to `{{...}}`, and do NOT move the footer outside the strip-and-reappend region (breaks idempotency).

**HTML-formatted, HTML-separated.** The signature MUST be wrapped in `<p>...</p>` (use `<br>` for in-sig line breaks). Separator is `<p>--</p>`, NOT `\n\n--\n`. Reason: Instantly's HTML sanitizer aggressively strips plain text and bare `--` outside element wrappers on PATCH round-trip — only `<a>` anchors and tag-wrapped content survive. An earlier plain-text signature was reduced to a stray `<a>distribute.you</a>` anchor on every PATCH (historic damage 2026-05-28: ~700 rows shipped with stacked broken anchors and no signature text). Guard: `should append HTML <p>--</p> separator + signature to body` + sibling cumulative-strip tests in `tests/unit/send.test.ts` assert the `<p>--</p>` form.

`stripAccountSignature` is HTML-tolerant. It looks for the EARLIEST occurrence of any of these standalone `--` markers (RFC 3676 sig delimiter, in its plain or HTML-wrapped forms):

- `\n\n--\n` plain (with optional trailing space, e.g. `\n\n-- \n`)
- `<p>--</p>` paragraph-wrapped (with optional `&nbsp;`)
- `<br>--<br>` line-break-wrapped
- `<div>--</div>` div-wrapped

…and slices everything from the marker onward. If no marker matches, body is returned unchanged. Senders whose original body legitimately contains one of these markers will lose content past that point on a re-send — accepted edge-case.

Historic bug 2026-05-28: the original `stripAccountSignature` matched only the plain `\n\n--\n` marker. Bodies stored as HTML never matched, so every retry-stuck re-send appended a fresh signature on top of the existing one. A row redispatched 72 times shipped a body with 72 stacked signatures. Fix: HTML-tolerant marker list + strip-then-append in `buildEmailBodyWithSignature`. Guard: this section + the `stripAccountSignature` and `buildEmailBodyWithSignature` cumulative-stack tests in `tests/unit/send.test.ts`.

If you add a new HTML wrapper form (e.g. `<section>--</section>`), add a regex to `SIG_MARKERS` AND a unit test case covering both the marker shape and a 3-stacked case for that shape.

## Send idempotency — reserve BEFORE the external Instantly call (DIS-148)

`POST /orgs/send` is idempotent under retry/concurrency for `(campaignId, leadEmail)`: a retried or concurrent send creates **at most one** Instantly campaign, and the loser returns **200 duplicate** (`{success:true, added:0, duplicate:true}`), NEVER a fatal 409. Trigger for the original bug: when instantly-service is slow, email-gateway's 10s `AbortSignal.timeout` fires and it retries — the abort only cancels the caller's wait, so the retry races the original. Both used to pass a read-only dedup check, both called Instantly (each picking the next sender in rotation → two real campaigns), and the loser's insert hit the unique index → fatal 409 → email-gateway mapped it 502 → Windmill flow failed.

**The fix is an ORDERING invariant in `src/routes/send.ts` — do NOT revert it to read-then-act.** The atomic claim on the `(campaignId, leadEmail)` unique index happens BEFORE `sendLeadToInstantly` (the external side-effect), not after:

1. **Reserve** — one `INSERT … onConflictDoUpdate(target:[campaignId,leadEmail], setWhere: reservation-is-stale).returning({id})`. The reserved row carries a unique **`reserving:<uuid>` sentinel** in `instantlyCampaignId` (column stays `notNull().unique()` — the sentinel is the "in flight" marker; NO nullable column, NO migration, so the 6 reader services that assume non-null are untouched). Winner ⇔ `RETURNING` non-empty. The single upsert covers all four cases: fresh-insert (winner), already-committed real-id row (loser → 200 dup), fresh concurrent sentinel (loser → 200 dup), and **stale** sentinel older than `STALE_RESERVATION_MS` (30s — crashed mid-send → reclaim → winner).
2. **Winner only** → `sendLeadToInstantly` → **phase-2** `UPDATE` overwrites the sentinel with the real `instantlyCampaignId`.
3. **Release on failure** — `sendLeadToInstantly` `!ok`, or any throw in the inner try → `releaseReservation()` `DELETE`s the row **only while it still matches `reserving:%`** (no-op once phase-2 committed) so a later legit retry can re-claim. Fail loud — no swallow.

`authorize` (credit affordability) + the leadId-conflict 409 stay BEFORE the reserve (they early-return without reserving). Cost provision/actualize (`addCosts`) is winner-only in the step loop — losers declare no cost. The leadId-conflict case (same email, different `lead_id`) is still a real **409**, unchanged. Guard: the "Reservation idempotency (DIS-148)" tests + the rewritten duplicate/concurrent tests in `tests/unit/send.test.ts`. This is idempotency-under-retry for the SAME `(campaignId, leadEmail)` — it does NOT dedup across *different* logical campaigns (that's the separate cross-campaign concern below).

**Platform sends (`campaignId=null`) reserve on `(run_id, leadEmail)`, NOT `(campaignId, leadEmail)` — fixed 2026-06-27 (v0.49.2).** Postgres treats every NULL as DISTINCT in a unique index, so for a platform send the `(campaignId, leadEmail)` arbiter NEVER collides → the original DIS-148 reservation silently did NOT cover platform sends, and every email-gateway timeout-retry created a fresh duplicate Instantly campaign. Incident: outlets-service run `27acff0b` fired 80 platform price-request sends; retries produced **160 active campaigns / 80 leads** (each journalist queued for the 3-step sequence twice; 80 dups paused via Instantly + DB during remediation). The fix branches the reservation `onConflictDoUpdate` target in `send.ts`: `campaignId` present → `(campaignId, leadEmail)`; `campaignId` null → **partial unique index `instantly_campaigns_platform_run_lead_active_idx`** on `(run_id, leadEmail)` `WHERE campaign_id IS NULL AND status='active'` (migration `0020_platform_send_dedupe.sql`; partial indexes are hand-written SQL + journal entry, same convention as `instantly_events_one_shot_dedupe_idx` — NOT a drizzle `.where()`, documented as a comment in `schema.ts`; do NOT drop it on a `db:generate` diff). The email-gateway retry forwards the SAME `x-run-id`, so `(run_id, leadEmail)` is the stable idempotency key (verified: all 160 incident rows shared `run_id`). Scoped to `status='active'` so the index covers exactly the live reservation/retry window and builds cleanly against healed prod data (paused/completed historical duplicates are excluded). Edge case: a platform send with `campaignId` null AND `run_id` null (x-run-id is OPTIONAL on `/orgs/send` via `requireOrgId`) degrades to old behavior (no dedup) — acceptable because platform sends in practice always carry `x-run-id`. Guard: the platform-dedup tests in `tests/unit/send.test.ts` (`reserves on the (run_id, lead_email) arbiter` / `platform retry … is an idempotent duplicate`).

## Send cost lifecycle — a provisioned hold resolves by REAL send evidence (org + platform)

Each step of a send provisions two `sequence_costs` holds (`instantly-account-email-sent` + `instantly-domain-email-sent`, `status='provisioned'`) at `/send` (and on retry-stuck redispatch). A hold MUST later resolve: **actualize** when the step's `email_sent` webhook fires (real spend), or **cancel** when the sequence can no longer send that step (reply/bounce/unsub, or the Instantly campaign is paused/completed). The live resolvers are `handleEmailSent` (actualize) + `cancelRemainingProvisions` (cancel) in `silver-promote.ts`, fired from `promoteEvent`.

**A hold is matched to its send by the per-lead `instantly_campaign_id`, NOT the caller `campaign_id` — persist-at-write (migration 0027).** The pre-0027 resolvers matched `sequence_costs.campaign_id = <caller id>` and early-returned on a null caller id. Platform sends store `campaign_id=NULL` (mig 0017), so the `campaign_id = NULL` predicate NEVER matched and `handleEmailSent` early-returned → **every platform-send email cost stranded `provisioned` forever** (560 rows at the fix; oldest 2026-06-09). The webhook/reconcile side identifies a send by its per-lead `instantly_campaign_id` (globally unique, 1 Instantly campaign = 1 lead), which was never stored on the hold — the classic "writer already held the value, persist it" gap (same family as mig 0025 `account_email`). Fix: `sequence_costs.instantly_campaign_id` column, populated at send time (`send.ts` phase-2 + `retry-stuck` redispatch — the id is already in hand), and the resolvers match on it via `matchesHoldCampaign` (`silver-promote.ts`): prefer `instantly_campaign_id = campaign.instantlyCampaignId` (works for org AND platform); fall back to `(instantly_campaign_id IS NULL AND campaign_id = campaign.campaignId)` for historical ORG rows (persisted pre-0027). The guard is now `if (!campaign.instantlyCampaignId) return` (always present). Do NOT revert the match to the caller `campaign_id` — it re-breaks platform sends. Guard: the "actualizes a PLATFORM send hold (campaignId NULL)" test in `tests/unit/silver-promote.test.ts`.

**Backlog reconcile — `POST /internal/campaigns/reconcile-provisioned-holds` (`src/lib/reconcile-provisioned-holds.ts`).** Unified, evidence-based sweep that supersedes the two single-purpose sweeps (`actualize-orphaned-sends` = the actualize case for org; `refund-stranded-holds` = the cancel case for org finished campaigns — both JOIN `sc→ic ON (campaign_id, lead_email)` and exclude `campaign_id IS NULL`, so NEITHER covered platform or lead-stopped-but-active). It classifies EACH provisioned hold from instantly-service's OWN real (`inferred=false`) silver evidence — NEVER an assumption (the observation that holds sit on `completed` runs is misleading: `send.ts` marks the step run `completed` at dispatch time regardless of whether the email fired):
- real `email_sent` at the hold's step → **actualize** (billable; `hasSent` wins even if the lead later stopped — the email already went out).
- else terminal (lead replied/bounced/unsubscribed on this send, OR its Instantly campaign is paused/completed, OR — for an unresolvable historical platform hold — the lead has no `active` platform campaign left) → **cancel** (refund; will never send that step).
- else in-flight (active campaign, step not yet dispatched) → **SKIP**. Cancelling here would under-bill if the step later sends — this is the legitimate sending-forecast pipeline (`pendingSends`), NOT stranding.

It acts on each hold's OWN `run_id` + `cost_id` (`updateCostStatus`), so it covers org AND platform, resolved AND historical rows with no campaign match. **The classification query MUST resolve the OWNING `org_id`/`user_id` for every hold — the runs-service PATCH authorizes on org scope, so a wrong/`'system'` identity is REJECTED and the hold strands as `transient`, healing nothing (this bit v0.62.1: the sweep read org from the caller-campaign join, which is NULL for platform sends → every platform actualize rejected while the sweep looked "successful"). Resolve via a LATERAL fallback to the lead's own campaign row when the caller-campaign join is empty; before trusting a run, verify the plan has ZERO null-org holds against prod (a unit test mocking identity cannot catch a NULL-from-join).** Idempotent (only `status='provisioned'`), resumable, in-cluster. Pure classifier `classifyHold(evidence)` is unit-tested (`tests/unit/reconcile-provisioned-holds.test.ts`); the route does IO. Body `{ dryRun?: boolean (default TRUE), limit?: number }`: dryRun returns the plan counts SYNCHRONOUSLY (read-only), `dryRun:false` → 202 + background mutation (log `reconcile-provisioned-holds: done`). **Wired into the daily reconcile cron (`reconcile-cron.yml`, commit mode)** as the ongoing self-heal — a transient runs-service error during a live actualize, or a platform send that stops, is drained next run. ## Account selection — lifecycle-gated (read from silver)

`sendLeadToInstantly` picks the sending account in two stages (`src/lib/send-lead.ts`):

1. **`fetchInProductionAccounts`** — the eligible pool, read PURELY from silver `instantly_accounts` (NO live `listAccounts` on the send hot-path). An account is sendable ⇔ `lifecycle_status == 'in_production'`. The lifecycle is auto-derived (health==100 AND delivery==100, not a brand/Instantly-deactivated domain) and maintained by `reconcileLifecycle` — see [Account lifecycle](#account-lifecycle--auto-driven-per-account-state-bsg). There is NO `MIN_WARMUP_SCORE` constant, NO `BLOCKED_DOMAINS` constant, and NO manual flag any more. If the pool is empty, `sendLeadToInstantly` returns `no_healthy_accounts_available` (a 500 upstream) — **NO silent fallback to weaker senders**.
2. **`pickCapacityAwareAccount(accounts, capacityByEmail)`** — CAPACITY-AWARE daily-limit fill: picks the account that can absorb one more email SOONEST under its own daily limit (MDL), NOT merely the globally least-loaded one. Per account, from the send-selection snapshot (`fetchAccountCapacityCached` in `account-sending-stats.ts`, pure buckets `aggregateQueueCapacity` in `queue-breakdown.ts`):
   - `MDL` = `account.daily_limit` (fallback `IN_PRODUCTION_DAILY_LIMIT=50`); `S0` = sentToday (real dispatches today); `Q0-first` = **count of never-contacted SEQUENCES** (1 first-email each ≈ today — NOT their `firstUnsent` step total, which would over-inflate today by the whole future sequence); `Q0-next` = followup steps projected today/overdue; `Q1-next` = followup steps projected tomorrow. `todayOcc = S0 + Q0-first + Q0-next`; `tomorrowOcc = max(todayOcc − MDL, 0) + Q1-next` (today's overflow `O1` + tomorrow's due). `Q0-next`/`Q1-next` reuse the EXACT per-step chained-delay projection + UTC-day bucketing as the ops queue breakdown (`classifyQueuedStep`), so "today"/"tomorrow" mean the same thing on both surfaces.
   - **Policy — first tier with a candidate wins, argMIN + random tie-break within:** (1) accounts with `todayOcc < MDL` → argMIN `todayOcc` (fill emptiest today); (2) else accounts with `tomorrowOcc < MDL` → argMIN `tomorrowOcc` (soonest room tomorrow); (3) else → argMIN `totalQueue` (globally emptiest queue). **Ties at the minimum → uniform random over ONLY the tied set** (never a heavier account), so a burst spreads across equally-preferred accounts. An account absent from the snapshot is all-zeros ⇒ maximally preferred. This REPLACED `pickLeastLoadedAccount` (single `load = queued + sentToday` metric, ignored the daily cap + when queued mail is due) which itself replaced the warmup-weighted random.
   - **Capacity snapshot is cached 60s per replica** (`fetchAccountCapacityCached` reuses the `/stats` TTL cache, `getOrSetCachedStats`): the two fleet-wide aggregations (email_sent-today scan + the queued-sequence loader) would saturate the 0.25-1 CU Neon compute under a send burst, same flood the stats cache guards. Do NOT switch to a live read per send.
   - **Correctness depends on queue attribution being known at `contacted`, not `sent`.** The chosen account is PERSISTED on the campaign row (`instantly_campaigns.account_email`, migration `0025`) at send time — `send.ts` phase-2 UPDATE + retry-stuck redispatch. The queued-sequence loader (shared by the capacity snapshot AND `fetchQueueBreakdownByAccount`) attributes via `COALESCE(persisted account_email, observed email_sent account)` with a **LEFT JOIN** so a lead counts against its account the instant it is `contacted` — no dependency on the lagging first `email_sent` webhook. Do NOT revert to observed-only attribution.

Guard: the `pickCapacityAwareAccount` (all 3 tiers + Q0-first-weighting + per-account-MDL + random tie-break) + capacity `POST /send` tests in `tests/unit/send.test.ts`, `tests/unit/account-sending-stats.test.ts` (`fetchAccountCapacity` merge + 60s cache), and `tests/unit/queue-breakdown.test.ts` (`aggregateQueueCapacity` buckets).

Primeforge (`X-Primeforge-Key` on `mcp.salesforge.ai/mcp`, or the `@salesforge/forge-cli` `forge primeforge …`) provisions real Google Workspace mailboxes. Import to Instantly uses the mailbox's **`appPassword`** field (16-char Google app-password, spaces stripped) + hosts **`imap.gmail.com`/`smtp.gmail.com`** (the CSV export's `password` = the login password and `imap.google.com`/`smtp.google.com` hosts are BOTH wrong → `IMAP connection failed`).

**Full mailbox list IS retrievable — hit the DIRECT REST API, not the CLI/MCP wrapper (2026-07-10 correction).** The prior note claimed a "hard 10-cap, no pagination, >10 not retrievable" — that is TRUE only for the **forge CLI / MCP `primeforge_list_mailboxes`** (it silently ignores `offset`/`limit`/`domainId`/`search` and returns the first 10). The underlying **direct REST** `GET https://api.primeforge.ai/public/mailboxes?workspaceId=<wks>&limit=100&offset=<N>` (header `Authorization: <PRIMEFORGE_API_KEY>`, raw, NO `Bearer`) paginates correctly and returns ALL mailboxes (verified: 35 mailboxes across 10 domains fetched via `limit=100`, and `offset=10&limit=50` returns rows 10-34). Each row carries `address`, `appPassword`, `status`, `domainId`. So when you need app-passwords for >10 mailboxes, use direct REST — do NOT fall back to the dashboard. Auth key = `PRIMEFORGE_API_KEY` from the `primeforge` skill's `secrets.env` (the `MAILFORGE_API_KEY` 401s on Primeforge). `export_mailboxes` is still broken via the MCP/CLI (needs a nested `platform`+`email`+`password` body the wrapper doesn't pass — use the paginated list instead). `bulk_dns_update` only does DMARC/forwarding (no custom-record add, so no CTD CNAME via API).

**Reconnecting a broken Instantly Google account — diagnose the `status_message` FIRST, don't guess (2026-07-14).** A Primeforge Google account shows as broken in Instantly with two DIFFERENT states that need OPPOSITE fixes — read `GET /api/v2/accounts/{email}`'s `status_message` (`{code, responseCode, response}`) before touching anything:
- **`status:-1` "Needs review" + `autofix_failed:true` = broken Instantly↔Google OAuth link.** The `PATCH imap_* + resume` trick (IMAP creds via Primeforge `appPassword` on `imap.gmail.com`/`smtp.gmail.com`) only flips it to `1` MOMENTARILY then it reverts — Instantly keeps the account `provider_code 2` (an attempted `provider_code` PATCH is accepted-but-ignored, stays 2) and keeps re-testing the dead OAuth. The DURABLE fix is the **native Primeforge → Instantly export** (Primeforge UI: Mailboxes → Export → Instantly, entering the Instantly LOGIN email+password + the case-sensitive workspace name — ours is `GrowthAgency.dev`). That export re-authorizes the Google domain-wide delegation (Instantly's OAuth Client-ID `536726988839-…apps.googleusercontent.com`) server-side. The Primeforge `export_mailboxes` API needs those same `platform`+`email`+`password` login creds (NOT the API key), so it can't be driven with the Instantly API key alone — it's a UI/login-cred action. There is NO Instantly API endpoint to re-mint Google OAuth (probed `/reconnect`,`/reconnect-oauth`,`/reauth`,… all 404); OAuth can only be re-granted by the domain's Workspace admin (= Primeforge for these domains).
- **`status:-3` + `status_message.responseCode:550` `"550-5.4.5 Daily user sending limit exceeded"` = Gmail throttling, NOT a connection/OAuth/export problem.** Instantly deactivates the account because Google rejects the sends. This is caused by OUR sending volume exceeding Gmail's per-user daily limit on (newish) Workspace mailboxes — do NOT chase OAuth/export fixes, and do NOT open a Primeforge refund ticket for it (the raw 550 proves it isn't their export). Fix = lower per-account volume so total (campaign `daily_limit` + `warmup.limit`) stays under the Gmail cap; the block auto-clears after ~24h of reduced sending. 2026-07-14 incident: the whole pre-warmed fleet (~15 accounts) escalated `-1 → -3` on this 550; remediation was capping every Google (`provider_code 2`) + IMAP-Mailforge (`provider_code 1`) account to `daily_limit:45` + `warmup.limit:5` (= 50/day, was 60) via `PATCH /accounts/{email}` (merge `warmup` to preserve `increment`/`advanced`). Repeated `resume` nudges make throttling WORSE — stop nudging once you see the 550.

## Stop-on-click — pause when brand maximizes signups (v0.50.1)

On a REAL `email_link_clicked` event, if the campaign's brand has `current_goal === 'signup'` (read LIVE from brand-service `GET /internal/brands/:brandId/runtime-context` — the goal is a MUTABLE runtime value, so a send-time snapshot would freeze a value meant to move), the lead's Instantly campaign is PAUSED. The prospect is on the landing page; the signup conversion happens there, so more cold emails only distract.

- **Fired as a fail-soft side effect from `promoteEvent` (`silver-promote.ts`), on non-inferred click events only.** Pure helpers + IO glue live in `src/lib/stop-on-click.ts` (`isStopOnClickEnabled`, `anyGoalIsSignup`, `maybeStopOnClickForSignup`); brand-service read in `src/lib/brand-client.ts` (`getCurrentGoal(s)`). Guard: `tests/unit/stop-on-click.test.ts` + `tests/unit/brand-client.test.ts`.
- **MINIMAL BY DESIGN — the side effect ONLY pauses on Instantly. Do NOT add cost-cancel / contact-delete / local-terminal-mark inline, and do NOT write local `status='paused'` here.** The existing machinery handles everything downstream, for free: (a) retry-stuck's live-status preflight sees the paused Instantly campaign and SKIPS redispatch (never resurrects it); (b) the nightly reconcile discovers the paused Instantly status → its `finish` closure cancels the lead's remaining provisioned holds (credit refund), deletes the contact (quota reclaim, if `DELETE_FINISHED_CONTACTS_ENABLED`), and marks the local row terminal. **If you mark the local row terminal in the webhook, `reconcileAll` SKIPS it → the finish closure never runs → the provisioned holds leak forever** (same class as the pre-fix stranded-holds bug). Leave the local row `active`; let reconcile discover and heal it.
- **Fail-soft, never throws into the webhook.** Any brand-service / key-resolution / Instantly-pause error is swallowed + logged (`console.warn`, "sequence continues"). A thrown error would 5xx the webhook and Instantly auto-pauses webhooks after repeated 5xx.
- **Multi-brand: ANY brand in `signup` mode triggers the stop** (`brand_ids` is an array; signup-max is the aggressive stop; most sends are single-brand).
- **Always-on — NO env kill-switch** (the `STOP_ON_CLICK_ON_SIGNUP_ENABLED` flag was dropped in v0.50.2, on user request — no such var in Railway). Runs unconditionally, gated only by the brand's runtime goal (`current_goal === 'signup'`) and fail-soft brand-service availability: a click on a non-signup brand is a natural no-op, and any brand-service / key / pause error leaves the sequence running. Requires `BRAND_SERVICE_URL` + `BRAND_SERVICE_API_KEY` on Railway (set in prod).

## Forward positive replies to the agency inbox

Instantly paywalls its Unibox/CRM, so a positive reply can no longer be seen/actioned from the Instantly UI without paying. The V2 API still exposes replies AND Instantly's OWN qualification, so instantly-service owns the "surface positive replies" loop with **zero dependency on the paid CRM**. When Instantly qualifies an inbound reply positive/interested, silver promotion forwards the **full conversation thread** (every message in order, from/to/date, the lead email, the campaign) to `kevin@distribute.you` (`ADMIN_NOTIFICATION_EMAIL`) via the production transactional-email path (Postmark under the hood). Pure logic in `src/lib/forward-positive-reply.ts`; unit-tested `tests/unit/forward-positive-reply.test.ts`.

- **Trust Instantly's OWN qualification — do NOT build a separate sentiment classifier.** "Positive" = exactly the events that flip `reply_classification='positive'` in silver-promote's `REPLY_CLASSIFICATION_MAP`: `lead_interested` / `lead_meeting_booked` / `lead_closed`. `POSITIVE_QUALIFICATION_EVENT_TYPES` mirrors that set and a unit test asserts the two never drift. `lead_meeting_completed` (interest 3) is deliberately EXCLUDED (not `positive` in the map). Do NOT add a heuristic/LLM classifier and do NOT widen the set without updating the map + the lockstep test.
- **Fired as a fail-soft side effect from `promoteEvent` (`silver-promote.ts`), on REAL (non-inferred) events only** — mirrors stop-on-click. Both the webhook path (`event_type=lead_interested…`) and the reconcile-poll path (`lt_interest_status` → the same event types) converge on it, so a positive qualification observed by EITHER ingestion path forwards once.
- **Exactly-once via an atomic claim on `instantly_campaigns.positive_reply_forwarded_at` (migration `0028`), NOT a webhook-dedup assumption.** The first positive event claims the column (`UPDATE … SET positive_reply_forwarded_at=now() WHERE instantly_campaign_id=$1 AND positive_reply_forwarded_at IS NULL RETURNING`) BEFORE the external send; a webhook retry, a reconcile re-poll, OR a re-qualification (interested→meeting_booked→closed each promote as distinct positive events) find it non-null and no-op. This is load-bearing because the SAME positive signal legitimately arrives via multiple paths + re-qualifications — a `COUNT`/webhook-first-insert guard alone would forward 2–3×. Do NOT replace the claim with "promoteEvent only fires on first insert" (that dedups a single event_type, not the cross-path/re-qualification case).
- **Claim released on send FAILURE (`positive_reply_forwarded_at = NULL`) so a later retry re-attempts** — at-most-once send, biased against the duplicate (the explicit no-go) while still self-healing a transient key/Instantly/email error. On send SUCCESS the claim stays. Do NOT release on success.
- **Thread = `GET /emails?campaign_id` (1 campaign = 1 lead = 1 thread), rendered PLAIN TEXT into `<pre>{{thread}}</pre>`.** The Instantly email object carries `body:{text,html}` + `from_address_email` + `to_address_email_list` on both list + get. Bodies are tag-stripped (`htmlToText`) and rendered plain-text so the template's `{{thread}}` is robust whether or not the engine HTML-escapes. Empty thread (reply not yet synced to `/emails`) still forwards the notification (lead + campaign + qualification) rather than blocking. The `positive-reply-forward` template is deployed at startup in `index.ts`.
- **Fail-soft, never throws into the webhook** (a 5xx makes Instantly auto-pause webhooks). No cost/runs declaration — same as the `campaign-error` admin notification (transactional-email-service owns its own send).
- **Org-scoped v1: platform sends (`orgId` null — e.g. journalist price-requests) are a documented follow-up**, not handled (distinct key path). Reuses `TRANSACTIONAL_EMAIL_SERVICE_URL` + `TRANSACTIONAL_EMAIL_SERVICE_API_KEY` + `ADMIN_NOTIFICATION_EMAIL` — **no new env vars**.

## Cross-campaign duplicate audit — read-only

`POST /send` dedups on `(campaign_id, lead_email)` (`instantly_campaigns_campaign_lead_idx`), **not** on `(org_id, lead_email)` or `(brand, lead_email)`. So the same person reached by two **different** logical campaigns of the same org/brand creates two separate active Instantly campaigns — the same prospect gets double-contacted. DIS-77 healed this once (Phase A cancelled 6 same-wave dups, Phase B cancelled 51 re-contacts) but its **root-cause prevention in `POST /send` was never shipped**, so duplicates re-accumulate.

`npm run audit:dupes -- <orgId>` is the standing read-only check for this:

- **Duplicate fact = Instantly API, authoritative.** Lists all campaigns (active = status `1`), sweeps all leads via `POST /leads/list`, groups by email, flags any email in ≥2 active campaigns. The local DB is deliberately NOT consulted for the fact — it can be stale.
- **Brand/org = local DB, label only.** Instantly has no brand field; `instantly_campaigns.brand_ids` / `org_id` are joined on `instantly_campaign_id` purely to classify each collision:
  - `same-brand` — ≥2 active campaigns share a brand. True redundant outreach (the dangerous case).
  - `cross-brand` — ≥2 active campaigns, all brands known, none repeated.
  - `unknown-brand` — ≥2 active campaigns but a campaign has no DB row (DB gap; duplicate still real).
- Pure detection logic lives in `src/lib/audit-duplicates.ts` (`findDuplicateContacts` / `summarizeDuplicates`), unit-tested in `tests/unit/audit-duplicates.test.ts`. The script `scripts/audit-cross-campaign-dupes.ts` does only the Instantly + DB IO.
- **Read-only — never PATCH/pause/cancel.** This audits; it does not heal. The healing pass is `heal:dupes` (below); the deferred `POST /send` prevention is separate work.

The **heal** (`scripts/heal-pause-dupe-campaigns.ts`, `npm run heal:dupes`):

- For each email in ≥2 active campaigns, **keep the oldest by `timestamp_created`** (NOT `created_at` — the `/campaigns` list does not return `created_at`; relying on it silently falls back to id-sort and keeps the wrong campaign) and **pause** the rest via `POST /campaigns/{id}/pause`. Collapses retry stacks AND distinct logical campaigns → one active per person.
- Selection logic is pure + unit-tested in `src/lib/heal-duplicates.ts` (`selectCampaignsToPause`); the script does only Instantly IO.
- **Pause, not delete** — reversible. **Dry-run by default**; `--commit` to act; `--limit N` to batch (~110ms/call ⇒ ~75 min for the full ~41k backlog). Idempotent + resumable: re-runs re-sweep live state, so already-paused campaigns leave the active set and are never re-touched.
- **Instantly-only.** No cost refund / local-DB `delivery_status` write — `handleCampaignError` / runs-service live at `*.railway.internal`, unreachable from a local shell. Cost reconciliation is a follow-up (DIS-148).
- **Order matters:** deploy the retry-stuck fix (DIS-148) BEFORE running `--commit`, else the worker re-spawns duplicates faster than the heal clears them.
- **Running it locally against prod.** key-service lives at `key-service.railway.internal`, which only resolves INSIDE Railway's network — a laptop `railway run` shell gets `ENOTFOUND`. So set `INSTANTLY_API_KEY` directly (bypasses key-service; audits that key's workspace — the platform key audits the shared workspace) and let `railway run -s instantly-service` inject `INSTANTLY_SERVICE_DATABASE_URL` (Neon, publicly reachable) for the brand labels: `railway run -s instantly-service -- bash -lc 'export INSTANTLY_API_KEY=…; npm run audit:dupes'`. The brand-label DB lookup is chunked + uses `inArray` (drizzle expands `ANY(${jsArray})` into a ROW expression that trips Postgres' 1664-entry limit; never pass a large JS array to `sql\`= ANY(...)\``).

## Finished-contact cleanup — reclaim quota

Instantly's plan limit (e.g. 25,000 "uploaded contacts") counts contacts **currently stored** across all campaigns (active, paused, completed, draft), NOT a lifetime cumulative — **deleting leads from a campaign frees the quota** (~5-10 min sync; confirmed [help.instantly.ai/articles/7918680](https://help.instantly.ai/en/articles/7918680-uploaded-contacts-limits)). Two important facts: the delete MUST be **campaign-level** (`DELETE /leads {campaign_id, delete_list}`) — deleting from "Lists" does not free quota; and **each duplicate counts separately** (same email in 2 campaigns = 2 contacts), so the cross-campaign dupe backlog also costs quota.

Each of our sends is its own per-lead campaign, so once a campaign is **finished** its contact is deletable with no data loss — all engagement history lives in our **local silver DB** (`analytics.ts`/`status.ts` read silver, never Instantly live). "Finished" = Instantly status **paused (2)** (e.g. operator paused after the prospect replied off-Instantly — Instantly never saw that reply) or **completed (3)** (sequence ran out). **Active (1) is never touched. No pause grace period** (locked option A — a pause means "done"; if you ever pause-to-edit-and-resume, the contact gets deleted).

Two paths:

- **One-shot** `npm run cleanup:finished-contacts` — the standing backlog sweep. Pure selection in `src/lib/cleanup-finished.ts` (`selectContactsToDelete` / `countDeletions`, `FINISHED_STATUSES = {2,3}`), unit-tested in `tests/unit/cleanup-finished.test.ts`; the script `scripts/cleanup-finished-contacts.ts` does only Instantly IO (paginated `listAllCampaigns` + `listAllLeads`, then `DELETE /leads` per contact). Dry-run by default; `--commit`, `--limit N`, `--json`. Idempotent/resumable; a 404 delete (already gone) is counted, not fatal. Instantly-only — no local DB write.
- **Recurring** (reconcile) — see [Recurring finished-contact deletion](#recurring-finished-contact-deletion-reconcile) once shipped: the nightly reconcile reads each campaign's Instantly status (already fetched in Phase 0, previously discarded) and, for paused/completed campaigns, deletes the lead **after** the read phases (1-3) backfill all current state, then marks the local row terminal so reconcile stops re-polling it.

**Running it locally against prod** — same as `audit:dupes`/`heal:dupes`: set `INSTANTLY_API_KEY` directly (bypasses key-service, which is internal-DNS-only): `railway run -s instantly-service -- bash -lc 'export INSTANTLY_API_KEY=…; npm run cleanup:finished-contacts'`. Run the dry-run first and read the counts before `--commit`.

### Recurring finished-contact deletion (reconcile)

The nightly reconcile (`reconcileAll` → `reconcileOneCampaign`, `src/lib/reconcile.ts`) deletes finished contacts on an ongoing basis so quota doesn't re-accumulate. Decision logic is pure + unit-tested in `src/lib/finished-contacts.ts` (`tests/unit/finished-contacts.test.ts`); reconcile is the IO glue.

- **Signal, zero extra API calls:** Phase 0 already fetches `GET /campaigns/{id}` for `not_sending_status`. It now also reads `config["status"]` (`parseInstantlyStatus`) — the campaign lifecycle code (1 active / 2 paused / 3 completed). Previously fetched and discarded.
- **ORDERING INVARIANT (load-bearing):** the status is *read* in Phase 0 but the delete *executes last*, via the `finish(result)` closure wrapping every return of `reconcileOneCampaign`. So the delete happens AFTER Phases 1-3 (analytics + `/leads/list` + `/emails` backfill) have captured all current state — a late reply/bounce is never dropped by the delete. Do NOT move the delete into Phase 0.
- **What it does** (`deleteFinishedContact`): for a finished campaign (`isFinishedInstantlyStatus` → status 2 or 3), campaign-level `deleteLeads(apiKey, instantlyCampaignId, [leadEmail])` (frees quota), then sets the local `instantly_campaigns.status` to `completed` (3) / `paused` (2) (`localTerminalStatus`). The leadEmail comes from our own row (1 campaign = 1 lead), not a live fetch.
- **COST-CANCEL INVARIANT (load-bearing — prevents the provisioned-hold leak):** in the same `finish` closure, when `isFinishedInstantlyStatus(instantlyStatus)`, reconcile FIRST calls `cancelRemainingProvisions` (exported from `silver-promote.ts`) to cancel the lead's still-`provisioned` `sequence_costs` (refund the org), THEN deletes the contact. A finished (paused/completed) campaign never sends its remaining steps and emits no `SEQUENCE_STOP_EVENT` for the un-sent steps, so without this cancel the provisioned holds leak forever (never actualized via `email_sent`, never cancelled, and the locally-terminal row is then SKIPPED by `reconcileAll` so never revisited). The cancel is **independent of `DELETE_FINISHED_CONTACTS_ENABLED`** (the credit leak exists whether or not quota is reclaimed) and idempotent (only touches `status='provisioned'`). Do NOT gate the cancel on the delete switch, and do NOT move it out of the post-read-phases `finish` closure (a real reply/bounce backfilled in Phases 1-3 must actualize/cancel its steps before this blanket cancel runs). Fixed 2026-06-27 (PR #334, issue #335) after prod showed ~12k provisioned rows stranded on paused/completed campaigns. The **pre-fix backlog** of already-terminal rows is NOT auto-healed (skipped by `reconcileAll`) — drained by the one-time **`POST /internal/campaigns/refund-stranded-holds`** endpoint (`src/lib/refund-stranded-holds.ts`): selects locally-terminal (paused/completed) campaigns that still carry `provisioned` holds and cancels them via the same `cancelRemainingProvisions`. Idempotent + resumable; mirrors `/reconcile` (202 + background, optional `{limit}`). Watch logs for `refund-stranded-holds: done`.

**Triggering `/internal/*` ops endpoints (reconcile, accounts-sync, refund-stranded-holds, actualize-orphaned-sends, placement-test/*) — POST the PUBLIC Railway domain from a LAPTOP, do NOT `railway ssh`.** These handlers make outbound internal-DNS calls (runs-service / key-service `*.railway.internal`) that only resolve inside the cluster — but the HANDLER runs in-container regardless of where the HTTP request originates, so `curl -X POST -H "x-api-key: $KEY" https://instantly.mcpfactory.org/internal/...` from a laptop works: the request hits the container, its `*.railway.internal` calls resolve server-side. Public domain = `RAILWAY_PUBLIC_DOMAIN` (`instantly.mcpfactory.org`); key = `INSTANTLY_SERVICE_API_KEY` (read the FULL value via `railway variables -s instantly-service --json` — the table view truncates to column width). `railway ssh -s instantly-service` is the WRONG tool for HTTP triggers: the container has **no `curl` and no `bash`** (`sh` only), and railway wraps ssh args through `sh -c` so a `node -e '…fetch(…)…'` breaks on the unescaped `(`. The genuine "must run in-cluster" cases are the LOCAL CLI scripts (`npm run audit:dupes` / `heal:dupes` / `cleanup:finished-contacts`) — those run node ON the laptop and need internal DNS themselves, so use `railway run -s instantly-service` or set `INSTANTLY_API_KEY` directly to bypass key-service. Verified 2026-07-08 (accounts-sync trigger, PR #423).
- **Skip terminal rows:** when deletion is enabled, `reconcileAll` filters out rows already locally terminal (`isLocallyTerminal` → status paused/completed) so a deleted contact is never re-polled — kills the nightly log noise. When disabled, every row is scanned (unchanged).
- **Kill-switch, default OFF:** gated by env `DELETE_FINISHED_CONTACTS_ENABLED` (`isDeleteFinishedEnabled` — exactly `"true"`, anything else incl. unset = OFF). Merge is safe with the var absent (reconcile stays read-only); enable after staging soak. Mirrors the `RETRY_STUCK_WORKER_ENABLED` convention.
- **Fail-loud, 404-tolerant:** a `DELETE /leads` 404 (lead already gone) is tolerated (`isLeadAlreadyGone`) — idempotent. Any other Instantly error propagates → the per-campaign wrapper counts it `failed` and retries next run. No cost/runs declaration (deleteLeads spends nothing).

**⚠️ LIVE IN PROD — `DELETE_FINISHED_CONTACTS_ENABLED=true` (production + staging, 2026-06-13, v0.42.0).** The recurring delete is ARMED. Operational consequence of locked **option A (no pause grace period)**: the instant a campaign is paused — including a **manual pause to edit-then-resume**, or an Instantly auto-pause (bounce/quota) — its contact is **deleted on Instantly at the next reconcile** (daily 03:00 UTC cron), irreversibly. Only pause when you mean "done". To disarm:

```
railway variables -s instantly-service --set "DELETE_FINISHED_CONTACTS_ENABLED=false"
```

## Inbox-placement history (Bronze/Silver/Gold) — per-account deliverability

**Feasibility (verified against the Instantly V2 API docs, 2026-07-01):** the V2 API exposes **NO standing per-account inbox-placement property**. The account object (`GET /accounts`) carries none. Placement exists ONLY as the output of an **inbox-placement TEST** — a point-in-time, subscription-gated (Growth Inbox Placement, else 402) event. `GET /inbox-placement-analytics?test_id=X` returns one row per **(test, sender, recipient)**: `is_spam` boolean + `recipient_esp` + SPF/DKIM/DMARC, **no per-account inbox/spam/missing percentage**. So the per-account figure is DERIVED by us, not read off the account. Do NOT re-add an "inboxPlacement from the account object" path — it does not exist.

**B/S/G pipeline** (`src/lib/placement-promote.ts` pure + `src/lib/placement-sync.ts` IO):
- **Bronze** (append-only): `instantly_placement_tests_raw` (one row/test), `instantly_placement_analytics_raw` (raw rows, deduped on Instantly's `analytics_id`).
- **Silver**: `instantly_placement_results`, PK `(test_id, account_email, recipient_esp)`. `aggregatePlacementRows` counts per (account, ESP): `inbox = received & !is_spam`, `spam = received & is_spam`, `seedTotal = distinct recipients`, `missing = seedTotal − inbox − spam`. Rows missing `sender_email`/`recipient_esp` are unattributable → skipped. **Per-ESP grain is deliberate** (Gmail-spam vs Outlook-fine is the whole deliverability finding — never blend it away in silver).
- **Gold**: `GET /internal/audit/account-health` injects the **latest test per account, blended across ESP** (`blendEspRows` — pooled counts, newest `testedAt`, null when never tested / seedTotal 0, never a fabricated 0%). `GET /internal/audit/account-health/history?email=` = per-test series. `buildAccountHealth(accounts, placementByEmail, sentTodayByEmail, queueSizeByEmail)` stays a pure mapper; the route does the silver read.

**Per-account throughput columns on `account-health` (#389 + #440, `src/lib/account-sending-stats.ts`, pure map in `account-health.ts`).** Per-account fields on the same gold row, all derived from OUR data or the Instantly account object, fail-loud, honest `0`/`null` — never fabricated:
- **`sentToday`** — `COUNT(*)` of REAL (`inferred=false`) `email_sent` silver events today (UTC) grouped by `account_email` (`fetchSentTodayByAccount`). The N in an "N/dailyLimit" read. Excludes inferred rows so it matches observed dispatches.
- **`sentYesterday`** (#440) — SAME query/provenance as `sentToday` (`fetchSentYesterdayByAccount`), window bounded to the full previous UTC day `[date_trunc('day',now)-1d, date_trunc('day',now))` so it excludes both today and older days. Honest `0` when the account sent nothing yesterday. When adding another day-window sent-count, mirror this exact bounded form — do NOT reuse the open-ended `>= today-midnight` shape of `sentToday`.
- **`warmupLimit`** (#440) — the account's daily WARMUP send volume (`account.warmup.limit`), read DIRECTLY off the account object, DISTINCT from `dailyLimit` (the max-send cap; e.g. warmup 10/day vs cap 50/day). **`GET /accounts` (LIST) DOES carry a `warmup` object per item — but ABBREVIATED to `{ limit }` only; the full `{ limit, increment, advanced }` appears solely on single-account `GET /accounts/{email}`** (why `setWarmupDailyLimit` GETs the single account before a warmup PATCH — to preserve increment/advanced). So `warmupLimit` needs no extra IO (it's on the list row), but do NOT read `warmup.increment`/`warmup.advanced` off the list — they are not there. `null` when Instantly reports no warmup config.
- **`queueSize`** — queued-but-not-sent step count per account (`fetchQueueSizeByAccount`): still-`provisioned` `sequence_costs` holds on active campaigns (reuses the EXACT `loadPendingLeads` gate — active + `delivery_status IN ('contacted','sent')`), collapsed to distinct un-sent steps, then attributed to the campaign's sending account via `COALESCE(persisted account_email, observed email_sent account)` (1 campaign = 1 lead = 1 account). The account is persisted on the campaign row at send time (migration `0025`), so a campaign pushed but **not yet sending IS attributed** from the `contacted` moment — the former "no observed account until first send" gap is closed. Only a historical row (NULL `account_email`) that also never emitted a real `email_sent` is unattributable → excluded, never fabricated.
- **`accountType`** — Instantly `provider_code` via `mapProviderCode` (1→`google`, 2→`microsoft`, 3/4→`imap`, else `null`). Guard: `tests/unit/account-health.test.ts` (`mapProviderCode` + injection cases) + `account-health-route.test.ts`.

**`queueSize` (pending STEPS) is PARTITIONED by a per-account queue BREAKDOWN over STEPS (#440 → #442 → step-partition, `src/lib/queue-breakdown.ts` pure + `fetchQueueBreakdownByAccount` IO).** Four `account-health` date buckets split each account's queued STEPS (every remaining un-sent email across all its queued sequences) by the projected send date of EACH step, and PARTITION `queueSize`: `queueSize === queuedFirstUnsent + queuedNextToday + queuedNextTomorrow + queuedNextLater` (the ops table's four bucket columns add up to the queued-STEPS total). A separate `queuedSequences` (a SEQUENCE / lead count, 1 Instantly campaign = 1 lead = 1 sequence) is ALSO exposed — a DIFFERENT granularity, kept side by side; only the bucket partition is per-step. **`queueSize` on the account-health row is sourced FROM the breakdown's step total (`breakdown.steps`) so the invariant holds by construction (same provisioned-step set); the send-path `fetchQueueSizeByAccount` is unchanged.** Inputs are all LOCAL: the distinct un-sent (provisioned) steps + last-sent step/timestamp from the cost ledger (`sequence_costs`), and the FULL per-step `delay` array from the campaign's LATEST bronze sequence config (`instantly_campaigns_config_raw.payload->'sequences'->0->'steps'`). **Delay indexing (load-bearing): cost steps are 1-based, the config `steps` array is 0-based, and the gap from cost-step `k` to `k+1` = `steps[k-1].delay` — resolved via the SHARED `delayForGap(fromStep, stepDelays)` helper exported from `sending-forecast.ts` (the same per-gap resolver + `STEP_GAP_CALENDAR_DAYS` fallback the fleet forecast uses; NOT a second copy). Projecting step `s` off last-sent `k` at `lastSentAt` CHAINS the hops: `projected(s) = lastSentAt + Σ_{hop=k..s-1} delayForGap(hop, stepDelays)` — every remaining step, not just the immediate next (the pre-step-partition bug bucketed only the next step, so the four buckets summed to SEQUENCES, not steps).**
- **The projected send date is a NOMINAL-CADENCE LOWER BOUND that COMPOUNDS across steps.** Instantly exposes no per-lead scheduled-send timestamp (only PAST timestamps + a business-hours window — probed live). We project `lastSentAt + Σ chained delays`; the ACTUAL dispatch slips LATER under daily-limit saturation / throttling / pauses (observed: a nominal 3-day gap actualizes at ~7.6d during the current backlog), and a step two hops out sums two nominal gaps so its lower bound is SOFTER than the next step's (the drift compounds). So `queuedNextToday` reads as "step DUE today-or-overdue", not "will certainly send today"; later steps in a sequence carry progressively softer dates. Honest framing — do NOT relabel as an exact schedule.
- **A never-contacted sequence (no first email sent) has NO `lastSentAt` anchor — all its un-sent steps are counted under `queuedFirstUnsent` ("not started yet"), never date-projected** (anchoring on "now" would fabricate a date for a sequence Instantly hasn't started).
- **v1 modeling (deliberate):** per-campaign non-sending-day windows (`campaign_schedule.days`/timezone) are NOT modeled — projection is the raw nominal UTC date (many campaigns send 7d/wk, so a weekday-only snap would be wrong here). Missing config → each hop's delay falls back to `STEP_GAP_CALENDAR_DAYS` (via `delayForGap`) so a step is never dropped from the partition, but **bronze config covers the entire projected set in practice (verified prod 2026-07-11: 4,926/4,926 contacted queued sequences have config, 0 null hop delays → fallback never fires).** An unattributable queued step (no persisted/observed account) is excluded, never fabricated — same as `queueSize`.
- **Coherence note (cadence RESOLVED v0.57.6, PR #445 + step-partition; DAY-BUCKETING coherence resolved by dropping the forecast weekend snap):** the fleet `sending-forecast` and this per-account breakdown share ONE cadence model — both CHAIN the same real per-step bronze delays via the shared `delayForGap` resolver (same path + 0-based indexing + `STEP_GAP_CALENDAR_DAYS` per-gap fallback) — AND now share ONE day-bucketing model: `scheduleLead` computes the SAME nominal date as `projectStepDate` (`lastSentAt + Σ delayForGap`), buckets it on the raw nominal UTC day with NO weekend snap, and clamps a past-due nominal slot to asOf — which mirrors the breakdown's `projKey <= todayKey ⇒ nextToday`. TWO former forecast-only behaviors caused the weekend disagreement (chart's today/tomorrow read 0 while this breakdown's `nextToday`/`nextTomorrow` counted the step) and were BOTH removed: (1) the weekend forward-snap (pushed a weekend-due step to Monday); (2) the base-clamp (`baseDate = max(lastSentAt, asOf)` then re-add the full gap — over-shot the nominal date whenever `lastSentAt < asOf`, i.e. always, and mis-bucketed overdue steps as future instead of today). With both gone, a step this breakdown counts as due today/tomorrow lands on the SAME UTC day in the forecast series → the forecast's per-day `scheduledCount` reconciles with the fleet-aggregated `nextToday`/`nextTomorrow`. Do NOT reintroduce a weekend snap or a base-clamp. The only remaining orthogonal difference: never-contacted leads (the forecast projects them from asOf; this breakdown keeps them in `firstUnsent`) — immaterial because the forecast's pending-lead gate is `delivery_status IN ('contacted','sent')`, so its leads are always contacted. Guard: `tests/unit/queue-breakdown.test.ts` (chained-projection + steps-sum invariant) + the breakdown cases in `account-health.test.ts` / `account-sending-stats.test.ts`, and the `sending-forecast.test.ts` cadence-coherence + raw-nominal-day + weekend-AC tests.

**The 4 states — `deriveLifecycle`, FIRST MATCH WINS (do NOT reorder):**
1. `domain ∈ instantly_domain_policy` (a DB table, NOT a code constant) → **`deactivated_by_user`**. Warmup 30/day, NO new sends, NEVER auto-promoted. ## Data layering — Bronze / Silver / Gold

Three layers, doctrine per `~/.claude/skills/data-layering/SKILL.md`:

- **Bronze** — raw external mirrors, append-only, never mutated. Tables: `instantly_webhook_payloads_raw`, `instantly_analytics_raw`, `instantly_emails_raw`, `instantly_leads_raw`, `instantly_campaigns_config_raw`. Each row = one payload from Instantly (webhook OR reconcile poll).
- **Silver** — canonical event log `instantly_events` + state row `instantly_campaigns`. Derived from bronze via `src/lib/silver-promote.ts`. Rebuildable.
- **Gold** — stats views in `src/routes/analytics.ts`, `status.ts`. Read silver only.

### Reply-sentiment gold = CURRENT sentiment (latest event per lead), NOT raw event counts

A reply's sentiment can be **re-qualified**: a webhook auto-classifies a reply `lead_interested`, then an operator manually re-qualifies it `lead_not_interested` via `POST /orgs/manual-qualifications` (which inserts a NEW silver event + pins `instantly_campaigns.reply_classification='negative'`, source `manual`). The silver event log keeps BOTH rows — that is the correct append-only audit trail, NOT a bug. So a lead's **current** sentiment is the LATEST of the 8 mutually-exclusive sentiment events per `(campaign_id, lead_email)`, manual winning ties — NOT every sentiment event ever recorded.

Gold reply-sentiment counts (`repliesPositive/Negative/Neutral`, `repliesDetail`) MUST therefore be derived from the latest-sentiment-per-lead, NEVER from `COUNT(*) FILTER (WHERE event_type = 'lead_interested')` over raw events. The raw-count form never drops the stale positive when a reply is later re-qualified negative, so model/brand/workflow positive-reply totals stay frozen after a manual reclassification (reported 2026-06-13). The 8 sentiment types are `SENTIMENT_EVENT_TYPES` in `src/routes/analytics.ts` (mirrors `REPLY_CLASSIFICATION_MAP` in `silver-promote.ts`).

- **Implementation:** `queryGroupedSentiment` / `querySentiment` (`analytics.ts`) run a `DISTINCT ON (campaign_id, lead_email) … ORDER BY timestamp DESC, (source='manual') DESC, created_at DESC, id DESC` CTE, then count by the winning type. `queryGroupedStats` + `queryStats` merge this into `repliesDetail`. Both routers (`analytics.ts`, `analytics-public.ts`) reuse these, so the fix covers model + brand + aggregate on both. `unsubscribe` stays an event count (separate signal, not a reclassifiable sentiment).
- **Pure gold fix — silver/bronze untouched.** The event log faithfully keeps the full reclassification history; only the gold *projection* reads the latest. Do NOT "fix" this by deleting/mutating silver rows on reclassification.
- **Per-step sentiment = current sentiment on the LAST step.** A reply is ONE outcome per lead for the whole sequence, not a per-step event. `queryStepSentiment` (`analytics.ts`) attributes each lead's CURRENT sentiment EXACTLY ONCE, to its **last reached step** = `MAX(step)` of the lead's `email_sent` events. So a reply re-qualified negative shows up only as negative on the last step — NEVER as a stale positive on the step where the original auto `lead_interested` event fired. Step-level *email* metrics (sent/opened/clicked/bounced/unsubscribe) stay genuinely per-step; only the reply *sentiment* is collapsed to the last step. Both routers' `stepStats` blocks use this. (Sequence stops on reply, so last sent step ≈ the auto reply step; for a manual negative Instantly never saw, it is simply the last step reached.)
- Guards: the latest-sentiment SQL-shape + re-qualified-reply unit tests in `tests/unit/stats.test.ts` / `stats-public.test.ts`, and the DB-backed `tests/integration/sentiment-reclassify.test.ts` (interested → manual not_interested ⇒ positive 0 / negative 1).

### Per-day campaign-fact gold (`contacted`/`notSending`/`cancelled`) — bucket the SAME rows the cumulative count uses

`contacted` is a campaign-table fact (one `instantly_campaigns` row = one lead pushed), NOT an event — cumulative non-day `contacted` = `COUNT(*)` over `instantly_campaigns` (`queryCampaignAggregates`). For `groupBy=day`, `queryGroupedCampaignAggregates` buckets those SAME rows by `localDayKey(c.created_at, tz)` (`campaignGroupColumn`), so **day-sum == cumulative `contacted` by construction**. `created_at` IS the contacted timestamp (CLAUDE.md: `firstContactedAt = MIN(c.created_at)`). Do NOT re-source per-day contacted from a different table (`instantly_lead_status_current.first_contacted_at` etc.) — a different row set risks count drift vs the cumulative number. **Day mode UNIONs aggregate-only day keys** into the groups (`queryGroupedStats`), so a day with `contacted>0` but zero email events still appears (all event metrics 0). Non-day groupings + per-day event metrics (sent/opened/clicked) are unchanged. Historic bug (v0.46.1): day mode hard-coded the aggregates to an empty Map → every day bucket showed `contacted=0` → dashboard "Pipeline activity" Outreach=0 despite real outreach. Guard: the groupBy-day tests in `tests/unit/stats.test.ts` (3-call order [events, aggregates, sentiment]; day-sum==cumulative; contacted-but-no-events day surfaces).

**Emit the parameterized `localDayKey` fragment EXACTLY ONCE per query — name it `group_key` in a CTE, then reference the alias everywhere else. NEVER interpolate `${groupCol}` more than once (and NEVER inside a `--` comment).** For `groupBy=day`, `groupCol` is `localDayKey(...)`, which carries the `${timezone}` bind. drizzle's `sql` tagged template re-emits that bind on EVERY `${groupCol}` interpolation — **including ones textually inside a `--` SQL comment** (drizzle builds the param list from interpolations; it does not parse SQL, so a comment is NOT inert). Two distinct failure modes from re-interpolation:

- **`42803`** — `${groupCol}` in both SELECT (`$1`) and `GROUP BY` (`$N`): Postgres matches grouped columns by exact expression incl. param position, so the mismatched placeholders trip `column "c.created_at" must appear in the GROUP BY clause`.
- **`08P01`** — `${groupCol}` in a `--` comment: drizzle binds the extra param but Postgres strips the comment before execution, so the executed SQL has N placeholders while N+1 params are bound → `bind message supplies N+1 parameters, but prepared statement requires N`.

The `GROUP BY 1` ordinal fix (v0.46.2, ab85fdc) dodged `42803` but its own explanatory comment still interpolated `${groupCol}` → `08P01` → every `/orgs/stats?groupBy=day` 500'd → features-service pipeline-activity 502'd (v0.46.3). The robust fix (v0.46.3): all three grouped day queries — `queryGroupedStats` (events), `queryGroupedSentiment`, `queryGroupedCampaignAggregates` — now use the SAME CTE shape: `SELECT ${groupCol} AS group_key ...` once inside a CTE, then `GROUP BY group_key` on the alias. No inline repetition, no ordinal hack, no comment interpolation. Non-day groupings keep the inline form (`groupCol` is `sql.raw(col)`, param-free, byte-identical every interpolation). Unit tests mock `db.execute` so they CANNOT catch param-drift at runtime — the guard is the static-SQL assertions in the day tests (`aggregateSqlText` contains `AS group_key` / `FROM grouped_campaigns c` / `GROUP BY c.group_key`, and `not.toContain("GROUP BY 1")` / `not.toContain("GROUP BY TO_CHAR")`). Any NEW grouped day query needs the same CTE shape + assertions.

### Silver inference (synthetic predecessors)

When a "downstream" event lands (e.g. `email_opened`) without a preceding "upstream" event (e.g. `email_sent`), silver promotion synthesizes the missing predecessor deterministically. Rules:

| Trigger | Inferred predecessors |
|---|---|
| `email_opened` step N | `email_sent` step N |
| `email_link_clicked` step N | `email_opened` step N + `email_sent` step N |
| `reply_received` step N | `email_sent` step N |
| `email_bounced` step N | `email_sent` step N |
| `lead_unsubscribed` step N | `email_sent` step N |
| `email_sent` step N | `email_sent` steps 1..N-1 (sequence cascade) |

Synthetic rows carry `inferred=true`, `source='inferred'`, `inferred_from_event_id` (audit), and `inferred_rule` (rule name). Timestamp = trigger event timestamp.

**One-shot upgrade:** for at-most-1-per-step event types (`email_sent`, `email_bounced`, `lead_unsubscribed`, `reply_received`), partial unique index `instantly_events_one_shot_dedupe_idx` enforces that. When a real webhook arrives after inference projected the event, the synthetic row is upgraded in place (`inferred=true → false`, real timestamp wins). Side effects fire because this is the first time the real signal is observed.

**Side effects on inferred events:** SKIPPED. Synthetic events are projection-only. `delivery_status` / `cost lifecycle` / `reply classification` update only when real external signals arrive.

**Backfill:** `npm run backfill:inferences` re-projects predecessors for every existing real silver event. Idempotent. CLI only — must not run in boot path.
